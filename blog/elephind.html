Recovering What LLM Agents Forget: Persistent Conversation Memory via Lightweight Retrieval

April 14, 2026

How a simple BM25 retrieval layer recovers a large share of information lost when LLM agents compact their context.

The Problem: Context Compaction Destroys Information

LLM-based agents maintain a running conversation with the user across hundreds of turns. But transformer context windows are finite. When the conversation grows too long, some agents resort to compacting — replacing older turns with a compressed summary to free up context space.

This is a lossy operation. The compaction summary preserves high-level themes and decisions, but drops specific details: exact configuration values, precise error messages, particular file paths discussed twenty minutes ago. When the agent later needs those details, they’re gone.

The question is: how much information is actually lost during compaction, and can a simple retrieval system recover it?

We built Elephind, a lightweight persistent memory system for LLM agents, and evaluated it against the LoCoMo benchmark — a standard academic dataset for long-term conversational memory. The results show that even a basic BM25 retrieval layer recovers a substantial share of information lost to compaction, with no embeddings, no vector database, and no fine-tuning required.

Approach: Append-Only Logs + BM25 Retrieval

Elephind’s design is deliberately minimal:

  1. Persist: Every conversation turn is appended to a plain-text Markdown log file, indexed by session.
  2. Detect: When the host agent signals a compaction event, Elephind ensures all turns are saved beforehand.
  3. Search: When the agent needs a detail from a compacted region, it calls elephind_search(query), which runs BM25 keyword search over all stored turns.
  4. Retrieve: The top-k matching turns are returned as formatted text, re-injecting the lost context.

No embedding models. No vector stores. No GPU. The entire system is ~500 lines of Python using rank_bm25, pydantic, and plain file I/O. Search latency is under 10ms per query.

Evaluation: The LoCoMo Benchmark

We use the LoCoMo (Long-term Conversational Memory) benchmark introduced by Maharana et al. at ACL 2024.1 LoCoMo is specifically designed to test whether systems can recall facts from long multi-session conversations.

Dataset

LoCoMo has 10 conversations, each roughly 600 turns and about 16K tokens, with the aggregate counts below.5

Each conversation spans 19–35 sessions between two speakers discussing their lives over several months. The conversations are grounded on time-aware event graphs and human-verified. Crucially, LoCoMo provides:

  • Per-session summaries written by the dataset authors — these serve as our compaction output.
  • 1,540 QA pairs across 4 categories, each with a ground-truth answer and evidence pointers to specific dialog turns.
  • Evidence dialog IDs identifying exactly which turns contain the information needed to answer each question.

This lets us directly measure whether retrieval recovers the evidence turns that compaction discarded.

In this post, we report category-level and aggregate metrics for single-hop, multi-hop, and open-domain QA. One additional LoCoMo QA type is omitted because its gold answers depend on calendar-date normalization in summaries rather than verbatim dialog phrasing.

Experimental Design

We compare three conditions:

ConditionAvailable ContextSimulates
VanillaSession summaries only (~20% of original text)Standard post-compaction state
ElephindSession summaries + top-k retrieved original turnsPost-compaction with retrieval
OracleFull original conversation textUpper bound (no compaction)

Vanilla represents what a standard LLM agent has after compaction: the summaries replace the original conversation. This is the baseline — no persistent memory, no retrieval.

Elephind adds retrieval on top: for each question, we run elephind_search(question) and append the top-k retrieved original turns to the summary context. This simulates an agent that uses Elephind to recall compacted details.

Oracle provides the full, uncompacted conversation. This is the ceiling — the best possible performance if no information were ever lost.6

Metric: Answer Containment Rate (ACR)

For each QA pair, we check whether the ground-truth answer can be found in the available context. We use a two-pass approach:

  1. Exact substring match — catches numbers, proper nouns, and precise phrases.
  2. Token overlap (75% threshold) — catches paraphrased answers where the key content words appear in the context even if the exact phrasing differs.

A QA pair is scored as “contained” if either pass succeeds. This is a conservative, fully automated metric that requires no LLM judge.

Top-k Sensitivity

Increasing k improves evidence recall, but we use k=5 as the default because it balances retrieval quality against context budget.

End-to-End LLM Evaluation

The ACR metric measures whether the right information is available in context. To measure whether a model actually uses that information to produce correct answers, we run a full end-to-end evaluation: Qwen2.5-7B-Instruct reads the context, generates an answer, and we score against the ground truth using standard SQuAD-style F1 and Exact Match (EM).

The LLM evaluation confirms the pattern seen in the ACR analysis: open-domain questions benefit the most (+15.2% F1, +10.9% EM), while single-hop questions see a small decrease. Overall, Elephind delivers a +38.0% relative improvement in F1 and a +63.8% relative improvement in Exact Match over the Vanilla baseline on the reported categories.

Notably, the single-hop category shows a small decrease. This happens because the additional retrieved turns can introduce noise for questions where the summary already contains a good answer — the model occasionally gets confused by extra context. This suggests that a smarter retrieval policy (retrieve only when the summary is insufficient) could further improve results, but even the naive “always retrieve” approach delivers strong overall gains.

Analysis

Where Elephind Helps Most

Open-domain questions see the largest gain (+18.3%). These questions ask about topics discussed in the conversation using distinctive vocabulary — exactly the scenario where BM25 excels. When the user asks “What did the charity race raise awareness for?”, BM25 matches “charity” and “race” against the original turn, even though the compaction summary might only mention “community events.”

Single-hop questions (+3.9%) benefit modestly. Many single-hop answers are already captured in the session summaries, limiting the room for improvement.

Multi-hop questions (+4.2%) see limited gains because answering them requires synthesizing information from multiple turns across different sessions — a single BM25 query may not retrieve all the pieces.

Reporting Scope

This post excludes the temporal LoCoMo QA type whose gold answers depend on explicit calendar-date normalization in summaries that is often absent from verbatim dialog turns. That mismatch can make uncompacted transcripts look artificially worse under string-containment metrics, so we focus on other categories where containment is a fair proxy for memory recovery.

We also note a February 2026 independent audit reporting 99 incorrect gold answers out of 1,540 (6.4%), which implies an effective dataset ceiling of about 93.6%.6 We expect this to have minimal impact on the relative Vanilla vs. Elephind gains reported here because both conditions are evaluated against the same labels.

BM25’s Vocabulary Gap

Evidence Retrieval Rate tops out at 52.8% even at k=20. The remaining 47% represents cases where BM25 cannot find the right turn due to vocabulary mismatch — the question and the evidence turn share no distinctive keywords. For example, a question asking “What hobby did Caroline pick up?” might not match a turn that says “I started painting watercolors last month.” Bridging this gap would require semantic search (embeddings), which we deliberately excluded from this study.

Compaction Summary Quality Matters

The LoCoMo session summaries are well-written — they capture major events, decisions, and factual details, averaging 20% of the original text length. Our Vanilla baseline of 47.7% on the reported categories reflects this quality. With lower-quality summaries (as might be produced by fast automatic compaction), we would expect the Vanilla baseline to be lower and Elephind’s relative advantage to increase.

Limitations

Answer matching in ACR is approximate. Our token-overlap metric is a lower bound — it misses cases where the answer requires inference rather than surface-level text matching. An LLM-based judge (as used in the original LoCoMo evaluation) would likely show different absolute numbers, though the relative gains should hold.

BM25 has a hard ceiling. Pure keyword search cannot handle vocabulary mismatch or questions requiring semantic understanding. Adding an embedding-based reranker or hybrid retrieval would address this, at the cost of additional infrastructure.

Dataset scale. LoCoMo contains 10 conversations with 1,540 QA pairs. While this is sufficient for detecting the effect sizes we observe, larger-scale evaluation would strengthen the conclusions.

Single-model end-to-end evaluation. We report answer-generation results with Qwen2.5-7B-Instruct only. Additional reader models and prompting strategies would improve confidence in generalization.

Reproducing These Results

The benchmark is fully self-contained and requires no API keys or GPUs:

git clone https://github.com/abouelhamayed/elephind.git
cd elephind
pip install -e ".[dev]"
python benchmarks/locomo_bench.py

The LoCoMo dataset is included in the repository under benchmarks/conversations/locomo10.json (CC BY-NC 4.0 license).5 Total runtime is under 30 seconds on a laptop.

Conclusion

A simple BM25 retrieval layer over persisted conversation logs recovers a substantial fraction of information lost during context compaction. On the reported LoCoMo categories:

  • Elephind achieves 61.5% Answer Containment Rate, up from 47.7% with summaries alone — a 29.1% relative improvement.
  • The system requires no embeddings, no vector store, no GPU, and searches in under 10ms.

The results suggest that for long-running LLM conversations, which tend to accumulate distinctive keywords like names, places, dates, and domain-specific terms, keyword-based retrieval is a surprisingly effective first line of defense against compaction-induced information loss. Semantic search would further close the remaining gap, but the complexity-to-value ratio of BM25 is hard to beat as a starting point.
